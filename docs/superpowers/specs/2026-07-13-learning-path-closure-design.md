# 学习主线闭环与发布成熟度设计

日期：2026-07-13

状态：已通过书面规格复核，进入实现计划

目标版本：MVP 之后的首个成品化迭代

## 1. 背景

项目已经完成 MVP 内容表面：10 个章节、8 个教学型 Demo、中英双语路由、时间线、谱系图和图源页面。当前静态构建生成 28 个页面，基线共有 32 个单元测试和 73 个 Playwright 测试，全部通过。

现阶段的主要问题不是内容数量，而是学习体验仍像一组相互链接的原型页面：学习顺序主要存在于首页，普通章节缺少统一的当前位置、完成反馈和上一章/下一章导航；总览页仍保留“Technical Closure”“MDX 章节可渲染”等内部验收文案；移动端步进 Demo 先显示步骤文字和按钮，核心图解出现在更下方，图上变化和解释容易脱节。

本迭代把项目从“可运行 MVP”推进到“可连续学习、可公开发布”的产品状态。

## 2. 目标与非目标

### 2.1 目标

1. 将 10 个章节组织成一条清晰、可自由跳读、可记录完成度的学习主线。
2. 在首页提供开始、继续和全部完成三种状态。
3. 在每章提供当前位置、上一章、完成本章和下一章入口。
4. 保持静态部署与渐进增强；没有 JavaScript 或本地存储时仍可阅读和顺序导航。
5. 让中英文页面共享同一份本地学习进度。
6. 缩短 Demo 内部重复标题，并改善移动端图解、步骤说明和控制按钮的视觉关联。
7. 移除面向开发者的原型验收文案，补齐相应回归测试。

### 2.2 非目标

- 不新增 Safety / Eval 章节。
- 不新增账号、后端、云同步、证书、积分、徽章或章节锁定。
- 不重做视觉系统，不引入新的颜色、圆角或卡片体系。
- 不在本轮扩充可下载图源，也不重写既有 Demo 机制。
- 不把“完成章节”与测验、滚动深度或 Demo 操作次数绑定。

## 3. 方案选择

本轮比较过三条路径：学习主线闭环、旗舰 Demo 精修、发布基础设施优先。选择学习主线闭环，因为它同时改善全部 10 个章节，并直接解决用户进入章节后缺少继续方向的问题。

旗舰 Demo 精修适合后续传播型迭代；SEO、性能和更完整的发布基础设施应在学习闭环稳定后继续推进。本轮只包含与学习闭环直接相关的移动端和可访问性修正。

## 4. 学习主线

学习主线使用以下稳定 ID 和顺序。顺序是产品契约，中英文只改变显示文案和路径前缀，不改变 ID。

| 序号 | ID | 中文路由 | 类型 |
| ---: | --- | --- | --- |
| 1 | `overview` | `/chapters/overview/` | 章节 |
| 2 | `search` | `/chapters/search/` | Demo |
| 3 | `expert-system` | `/chapters/expert-system/` | Demo |
| 4 | `bayes` | `/chapters/bayes/` | Demo |
| 5 | `decision-boundary` | `/chapters/decision-boundary/` | Demo |
| 6 | `cnn` | `/chapters/cnn/` | Demo |
| 7 | `attention` | `/chapters/attention/` | Demo |
| 8 | `llm-system` | `/chapters/llm-system/` | 章节 |
| 9 | `rag` | `/chapters/rag/` | Demo |
| 10 | `agent` | `/chapters/agent/` | Demo |

英文路径通过现有 `toLocalizedPath` 生成，例如 `rag` 对应 `/en/chapters/rag/`。

所有章节始终可访问。完成状态只用于反馈和“继续学习”，不用于解锁内容。

## 5. 用户旅程

### 5.1 首页

- 没有进度时：主 CTA 为“从总览开始”。
- 已完成部分章节时：展示“已完成 X / 10”，主 CTA 为“继续学习：{第一篇未完成章节}”。
- 全部完成时：展示学习主线完成状态，并提供时间线、谱系图和图源三个回顾入口。
- 有进度时显示低强调度的“重置学习进度”。点击后在原位置展开“确定重置？”与“取消”，不使用浏览器原生确认弹窗。

“继续学习”始终指向顺序中的第一篇未完成章节。即使用户跳到后面的章节并标记完成，也不会跳过更早的未完成章节。

### 5.2 章节顶部

每章标题之前或导语之前显示紧凑的主线位置：

```text
学习主线 · 第 3 / 10 章
```

同时显示一条细进度线。该进度线表达当前章节在主线中的位置，不声称用户已经完成之前的所有章节。可访问名称必须明确为“当前位于第 3 章，共 10 章”。

### 5.3 章节底部

在参考资料之后统一渲染章节收口：

- 第一章不显示上一章，其余章节显示“上一章：{标题}”。
- 非最后一章的主 CTA 是“标记完成并继续”，目标为顺序中的下一章。
- 已完成章节再次访问时显示“本章已完成”，并保留“继续下一章”。
- 最后一章完成后不自动离开页面，改为显示“学习主线已完成”以及时间线、谱系图和图源入口。
- 如果用户跳读并只完成了最后一章，而前面仍有未完成章节，则显示“本章已完成”和“继续未完成章节”，链接到顺序中的第一篇未完成章节；只有 10 章全部完成时才显示“学习主线已完成”。

完成动作是用户主动行为，不根据页面滚动、停留时间或 Demo 操作自动触发。

## 6. 信息架构与模块边界

### 6.1 `learningPath`

新增站点级模块 `apps/site/src/learning/learningPath.ts`，只负责稳定结构：章节 ID、基础路由、类型和顺序。它不读取浏览器状态，也不包含大段页面文案。

模块提供以下能力：

- 获取完整有序主线。
- 按 ID 获取当前章节。
- 获取上一章和下一章。
- 获取第一篇未完成章节。
- 校验 ID、路由和顺序唯一性。

### 6.2 学习主线本地化文案

新增 `apps/site/src/i18n/learning.ts`，集中维护：

- 10 个章节的中英文短标题。
- 首页进度、章节位置、完成、重置和存储失败文案。
- 8 个 Demo 的短活动标题，例如“RAG 流程演示”与“RAG Pipeline Walkthrough”。

短活动标题属于站点呈现，不加入 `packages/data` 的教学 manifest，避免改变既有数据拓扑和中文基线夹具。

### 6.3 `learningProgress`

新增 `apps/site/src/learning/learningProgress.ts`，作为唯一的本地存储适配层。UI 组件不得直接解析或写入 JSON。

```ts
interface LearningProgress {
  version: 1;
  completedChapterIds: string[];
}
```

存储键为 `ai-history-learning-progress`。适配层负责：

- 安全读取和规范化进度。
- 幂等地完成一个章节。
- 重置进度。
- 过滤未知、重复或非字符串 ID。
- 报告存储是否可用，而不是向 UI 抛出异常。

版本未知、JSON 损坏或结构错误时回退为空进度，但读取过程不覆盖原始值。未来需要迁移时必须显式增加版本迁移函数；本轮不猜测未知版本含义。用户下一次主动完成或重置时，适配层才写入有效的 v1 状态。

### 6.4 组件

- `ChapterProgress.astro`：静态渲染当前序号和位置进度，不依赖本地存储。
- `ChapterJourney.svelte`：读取完成状态，渲染上一章、完成/继续和全章完成状态。
- `HomeLearningProgress.svelte`：渲染首页开始、继续、全部完成和重置状态。

`StandardDemoChapter.astro` 统一接入 Demo 章节；`OverviewChapter.astro` 和 `LlmSystemChapter.astro` 分别接入两个非标准章节。路由文件不复制进度逻辑。

## 7. 渐进增强与状态流

服务器渲染必须提供完整的章节位置、上一章链接和下一章链接。非最后一章的完成 CTA 使用真实的下一章 `href`：

- JavaScript 可用时，点击先写入完成状态，再使用原生链接导航。
- JavaScript 不可用时，链接仍导航到下一章，但不保存进度。

首页服务器输出保持现有“从总览开始”基线。客户端挂载后读取本地进度，并在固定尺寸的进度区域内更新 CTA，避免明显布局跳动。

同一页面中的进度组件通过 `ai-history:learning-progress-changed` 自定义事件同步；其他标签页通过标准 `storage` 事件同步。语言切换继续使用同一存储键和章节 ID，因此不复制或迁移进度。

## 8. 存储失败行为

组件挂载时只通过安全访问和读取判断本地存储是否可用，不执行试探性写入。实际写入能力在用户首次完成或重置时确认，避免探测过程改动已有进度。

- 存储不可用时，首页和章节显示低强调度提示：“本设备无法保存学习进度，章节仍可正常阅读。”
- 非最后一章仍渲染真实下一章链接，文案改为“继续下一章（不保存进度）”。
- 如果挂载时可用、完成写入时意外失败，则第一次点击停留在当前页并显示提示；CTA 随即变为普通的下一章链接，第二次点击可以继续。
- 最后一章写入失败时不显示虚假的全章完成状态，只显示存储提示和三个回顾入口。

完成写入是幂等操作。重复点击或重复完成同一章节不会生成重复 ID。

## 9. Demo 成品化

### 9.1 标题

章节 H1 保留完整教学问题。`DemoShell` 中重复的同名长 H2 改用 `i18n/learning.ts` 提供的短活动标题。区域仍通过 H2 获得清晰的可访问名称。

### 9.2 移动端步进布局

桌面端保留“图解在左、步骤说明和控制在右”的现有布局。

在 `max-width: 900px` 下，`StepperDemo` 的顺序改为：

1. 图解；
2. 当前步骤说明；
3. 上一步/下一步控制。

目标是用户滚动到 Demo 后，在常见手机视口中同时看到当前图解、步骤标题和至少一个推进控制。`SvgScene` 继续默认“适配屏幕”，用户可以切换“放大查看”进行横向滚动。按钮不做固定或悬浮，避免遮挡正文和增加滚动状态复杂度。

该顺序只改变布局，不改变步骤状态机、SVG 节点 ID、动效或教学数据。

## 10. 内容清理

总览章节删除以下面向实现验收的内容：

- `Technical Closure`
- `MDX 章节可渲染`
- “这个页面验证 Astro + MDX 章节闭环”等实现说明

总览仍保留历史主线、阅读方式、简化说明和参考资料。删除内部文案后，不用新的营销卡片填充空间。

## 11. 可访问性

- 章节位置同时提供可见文本和语义化进度值。
- 完成状态和存储失败消息使用 `aria-live="polite"`，不抢占焦点。
- 上一章、下一章和重置确认控件保持至少 44px 高。
- 当前章节位置、已完成状态和错误状态不能只依赖颜色。
- 重置采用页面内二次确认，确认与取消均可键盘操作。
- 页面导航后使用浏览器默认的文档焦点与滚动行为；不强行把焦点移动到任意卡片。
- 最后一章完成后，将完成标题设为可编程聚焦，并仅在用户主动完成时聚焦该标题。
- 现有 `prefers-reduced-motion` 规则继续覆盖新增过渡；本轮不新增必须观看的动画。

## 12. 测试策略

### 12.1 单元测试

为学习路径和进度适配层增加测试：

- 章节 ID、基础路由和顺序唯一。
- 前一章/后一章边界正确。
- 第一篇未完成章节计算正确，包括乱序完成。
- 有效状态读取、幂等完成和重置。
- 损坏 JSON、未知版本、错误字段、重复 ID 和未知 ID 的规范化。
- `localStorage` 读取或写入抛错时返回不可用状态。

### 12.2 Playwright

增加以下行为覆盖：

- 首页的开始、继续和全部完成三种状态。
- 重置进度的二次确认、取消和确认。
- 10 个章节的当前位置、上一章和下一章契约。
- 完成章节后刷新页面仍保持状态。
- 中文完成后切换英文，状态和下一未完成章节保持一致。
- 存储不可用时显示提示，下一章链接仍可访问。
- 没有 JavaScript 时，章节仍包含可用的前后导航。
- 375px、390px 和桌面宽度下，移动步进布局满足图解、说明和控制的几何关系。

### 12.3 截图回归

只增加两组稳定截图，避免扩大维护成本：

1. 390px 首页“部分完成”状态。
2. 390px RAG Demo 初始步骤，画面包含图解、步骤说明和推进控制。

截图使用 Playwright 固定 Chromium、固定视口和确定性本地进度，不包含时间、随机数据或外部资源。

## 13. 验收标准

1. 10 个章节全部显示统一的当前位置和前后导航。
2. 首页能准确显示开始、继续和全部完成状态。
3. 完成状态在刷新和中英文切换后保持一致。
4. 本地存储不可用时没有未捕获异常，顺序导航仍可用。
5. 无 JavaScript 时所有章节仍可阅读，并能通过静态链接顺序浏览。
6. 移动端步进 Demo 按“图解 → 说明 → 控制”排列；RAG 和 Agent 的首个步骤通过几何回归，RAG 初始步骤另通过截图回归。
7. 页面不再出现内部原型验收文案。
8. 不改变既有 Demo 图拓扑、稳定 SVG ID、简化说明和参考资料。
9. `pnpm validate:data`、`pnpm lint`、`pnpm build`、`pnpm test` 全部通过。

## 15. 发布顺序

实现时按以下风险顺序推进：

1. 学习路径与本地状态纯逻辑及单元测试。
2. 静态章节位置和前后导航。
3. 首页与章节完成状态。
4. 存储失败、重置和跨语言行为。
5. Demo 短标题与移动端布局。
6. 内容清理、截图回归和完整验证。

每一步保持静态链接可用，不把整个发布依赖于最后一次大规模接线。
