# Multilingual Support Design

Date: 2026-07-10
Branch: codex/add-i18n

## Goal

Add bilingual support for the static AI history atlas with Simplified Chinese (`zh-CN`) and English (`en`). The site should choose a sensible language automatically for first-time visitors, let users switch languages manually, remember that manual choice, and keep all demos static, scripted, and backend-free.

## Scope

In scope:

- Keep existing Chinese routes as the default canonical routes, such as `/`, `/timeline/`, and `/chapters/rag/`.
- Add English mirror routes under `/en/`, such as `/en/`, `/en/timeline/`, and `/en/chapters/rag/`.
- Add a small local i18n layer for route mapping, language detection, page chrome, page copy, and package data access.
- Add a header language control that switches to the equivalent page in the other language and stores the user's choice.
- Make first-time root visits choose English when browser language strongly indicates English; manual choice always wins.
- Translate user-visible site UI, overview pages, chapter pages, demo copy, timeline, lineage, and LLM-system data for the first shipped language pair.
- Preserve Chinese behavior and tests as regression coverage.

Out of scope:

- Machine translation at runtime.
- Backend language negotiation.
- User accounts or server-side preferences.
- Translating external paper titles or stable technical identifiers where English is already the standard label.
- Adding new AI demo behavior.

## Recommended Approach

Use static bilingual routing plus typed local translation data.

Chinese remains available at the existing routes. English pages are generated from explicit `/en/` route files or shared page helpers that receive `locale: "en"`. This keeps the site static-deployable, makes each language addressable, and avoids client-only replacement of rendered HTML.

## Language Model

Define:

```ts
export const supportedLocales = ["zh-CN", "en"] as const;
export type Locale = (typeof supportedLocales)[number];
export const defaultLocale: Locale = "zh-CN";
```

Use helpers for:

- `normalizeLocale(value)`: maps browser/user values to `zh-CN`, `en`, or `undefined`.
- `getPathLocale(pathname)`: returns `en` for `/en/...`, otherwise `zh-CN`.
- `toLocalizedPath(pathname, locale)`: maps the current route to the matching route in another language.
- `getSiteCopy(locale)`: returns header, button, metadata, and common UI labels.

## URL And Detection Behavior

Route rules:

- Chinese: `/`, `/timeline/`, `/lineage/`, `/diagrams/`, `/chapters/.../`.
- English: `/en/`, `/en/timeline/`, `/en/lineage/`, `/en/diagrams/`, `/en/chapters/.../`.
- Existing Chinese links remain valid.
- English links keep users inside `/en/`.

Detection rules:

- Manual language selection is stored in `localStorage` and a cookie so future root visits can honor it early.
- Visiting `/` may redirect to `/en/` when no manual preference exists and `navigator.languages` prefers English.
- Visiting a specific content path, such as `/chapters/rag/`, should not unexpectedly redirect to English. The visible language switch is the explicit path for changing language.
- Visiting `/en/...` sets the page locale to English and should offer a switch back to the equivalent Chinese path.

This keeps auto-detection helpful without making deep links unstable.

## Layout And Language Switcher

`BaseLayout.astro` accepts:

- `locale`
- `title`
- `description`
- optional `localizedPath`

It renders:

- `<html lang="zh-CN">` or `<html lang="en">`.
- localized site title and metadata suffix.
- localized navigation labels.
- a compact language control in the header.

The language control:

- links to the equivalent route in the other language.
- sets manual preference before navigation.
- uses accessible text such as `中文` and `English`.
- does not require JavaScript for navigation, but uses JavaScript to persist preference.

## Content Organization

Add `apps/site/src/i18n/` with:

- `locales.ts`: locale types and route helpers.
- `siteCopy.ts`: shared chrome and common labels.
- `pages.ts`: home, timeline, lineage, diagrams, overview, and chapter page copy.
- `autoLanguage.ts` or inline layout script for preference and root detection.

Prefer page data objects over scattered conditionals. For example, a chapter page should read a localized object containing title, description, lede, notes, references label, and any action labels.

Keep Chinese explanatory copy near the page or data file that owns it by storing Chinese and English variants together in the same feature-oriented module.

## Data Package

Keep stable geometry and IDs shared. Localize only text fields.

Add locale-aware exports such as:

```ts
export function getRagPipelineDemo(locale: Locale = defaultLocale): PipelineDemo;
export function getAiTimelineEntries(locale: Locale = defaultLocale): TimelineEntry[];
export function getAiLineageNodes(locale: Locale = defaultLocale): LineageNode[];
```

Existing named exports can remain Chinese defaults for backward compatibility during migration:

```ts
export const ragPipelineDemo = getRagPipelineDemo("zh-CN");
```

This limits churn in existing consumers and tests while allowing English pages to opt into English data.

## Svelte Demo Components

Demo components should accept a `locale` prop and select localized data through `@ai-history/data` accessors, or accept a fully localized demo object when that is cleaner.

Common UI labels in `packages/demo-core` become props with defaults:

- `demoKicker`
- `learningGoalsLabel`
- `simplificationLabel`
- `previousLabel`
- `nextLabel`
- `fitLabel`
- `detailLabel`

Defaults remain Chinese to preserve existing behavior. English pages pass English labels from site copy or demo copy.

## MDX Overview Chapter

The current overview chapter is MDX and mostly static Chinese. For the first bilingual pass, create an English route that renders equivalent Astro/MDX content under `/en/chapters/overview/`. Avoid building a generalized MDX i18n system until more MDX chapters exist.

## Error Handling And Fallbacks

- Unknown or unsupported locale values fall back to `zh-CN`.
- Missing localized data should fail TypeScript checks by using typed translation objects with `satisfies Record<Locale, ...>`.
- If client storage is unavailable, language switching still works through links; only preference persistence is skipped.

## Testing

Add or update tests to cover:

- Chinese routes still render current canonical Chinese text and interactions.
- English home, timeline, lineage, diagrams, overview, and at least one interactive chapter render English text.
- Header language switch maps `/chapters/rag/` to `/en/chapters/rag/` and back.
- Manual switch stores preference.
- Root auto-detection sends English-preferring first-time visitors from `/` to `/en/`.
- `<html lang>` matches the rendered language.
- Demo controls and scenario selectors show localized labels.
- `pnpm build` passes.

Keep the existing Playwright smoke suite as the regression backbone.

## Acceptance Criteria

- `pnpm validate:data`, `pnpm lint`, `pnpm test`, and `pnpm build` pass.
- First-time English browser visit to `/` can land on `/en/`.
- Manual language selection overrides auto-detection and persists across root visits.
- All primary routes have working Chinese and English variants.
- Demos remain static and require no backend, API key, database, real model, or translation service.
- No existing Chinese route is broken or renamed.

## Implementation Notes

The safest implementation order is:

1. Add locale types, route helpers, and failing tests for path mapping and root detection.
2. Localize shared layout and add the language switcher.
3. Add English route mirrors for primary pages.
4. Localize `packages/data` manifests and demo-core labels.
5. Update Svelte demos to accept locale or localized copy.
6. Expand Playwright coverage and run the full validation suite.
